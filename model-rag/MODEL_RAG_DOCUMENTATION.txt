1.1 MODEL TYPE
    - Algorithm: RandomForestClassifier (scikit-learn)
    - Estimators: 100 trees
    - Max depth: Unlimited
    - Criterion: Gini impurity
    - File: model-rag/model.pkl

1.2 PURPOSE
    Predicts whether a solar inverter will experience a failure in the near
    future based on current telemetry readings. Binary classification:
      0 = No failure expected
      1 = Failure likely

1.3 DATASET
    - File: solar_processed_dataset.csv
    - Rows: 1,047,903 telemetry records
    - Columns: 29
    - Date range: March 2024 onward
    - Inverters: 6 (across 3 plants)
    - MACs:
        Plant 1: 54-10-EC-8C-14-6E (INV-6E), 54-10-EC-8C-14-69 (INV-69)
        Plant 2: 80-1F-12-0F-AC-BB (INV-BB), 80-1F-12-0F-AC-12 (INV-12)
        Plant 3: ICR2-LT1-Celestical-10000.73 (INV-LT1),
                 ICR2-LT2-Celestical-10000.73 (INV-LT2)

1.4 FEATURES (24 total)
    Power metrics:
      - inverter_power, pv1_power, pv2_power, grid_power
      - energy_today, energy_total

    Voltage & current:
      - pv1_voltage, pv2_voltage, pv1_current, pv2_current
      - power_factor

    Temperature:
      - inverter_temp, ambient_temperature, temp_difference

    Grid:
      - grid_frequency

    Derived:
      - smu_std_current, rolling_mean_power_24h, rolling_std_power_24h
      - day_of_week

    Categorical:
      - inverters_alarm_code, inverters_op_state
      - plant_id_plant_1, plant_id_plant_2, plant_id_plant_3

    Target variable: future_failure (True/False)
    Note: hour_of_day is in the dataset but NOT used by the model.

1.5 PERFORMANCE
    - Test accuracy: 95.88%
    - Train accuracy: 95.93%
    - No overfitting (0.05% gap only)
    - Class balance: 53.3% failure vs 46.6% no-failure (nearly balanced)
    - No data leakage: failure_label and future_failure correctly excluded

    Per-plant accuracy:
      Plant 1: 96.5%
      Plant 2: 96.9%
      Plant 3: 93.9%

1.6 KNOWN MODEL ISSUES
    - ambient_temperature is always 0 in the dataset (sensor issue). This
      makes temp_difference identical to inverter_temp. Both features are
      in the model, wasting 13.86% combined importance on the same signal.
    - hour_of_day is excluded from features despite being present in data.
    - These are minor issues. Overall model grade: A-

1.7 SHAP EXPLAINABILITY
    - File: model-rag/exp_model.py
    - Uses TreeExplainer for feature importance per prediction
    - Shows top 5 factors driving each prediction
    - Enables targeted maintenance instead of generic inspection

1.8 PREDICTION SCRIPT
    - File: model-rag/predict.py
    - Input: new_telemetry_data.csv (same 24 feature columns)
    - Output: Binary prediction + failure probability (0.0 to 1.0)

    F --> G[LLM Answer + Sources]

2.1 WHAT IS RAG?
    RAG combines information retrieval with LLM text generation. When a user
    asks a question, the system:
      1. Searches a knowledge base for relevant documents (retrieval)
      2. Passes those documents as context to an LLM (augmentation)
      3. The LLM generates an answer grounded in the retrieved context

    This prevents hallucination and ensures answers are based on actual
    Sunlytix fleet data, alarm codes, and maintenance procedures.

2.3 RAG FILES

    model-rag/generate_knowledge_base.py
      - Reads solar_processed_dataset.csv
      - Generates 30 diverse documents covering:
          Section 1:  System overview (fleet stats, plant/inverter mapping)
          Section 2:  Per-inverter profiles (6 profiles with failure stats)
          Section 3:  Per-plant summaries (3 plants)
          Section 4:  Alarm code reference guide (10 alarm codes with counts)
          Section 5:  Operational state reference (10 op states with counts)
          Section 6:  Feature interpretation guide (all 24 features explained)
          Section 7:  Root cause analysis (6 failure modes with symptoms/actions)
          Section 8:  Maintenance schedule (daily/weekly/monthly/quarterly/annual)
          Section 9:  Threshold guidelines (temp, power, grid, current, PF)
          Section 10: Data-driven failure patterns (pre-failure vs normal stats)
          Section 11: Temporal patterns (hourly, weekly, seasonal)
          Section 12: Case studies (real high-temp and comm-failure events)
          Section 13: ML model interpretation guide
          Section 14: FAQ documents (6 common questions with detailed answers)
          Section 15: Comparative analysis across plants
      - Output: knowledge_base.txt (documents separated by ---DOC---)

    model-rag/create_chunks.py
      - Reads knowledge_base.txt
      - Splits by ---DOC--- separator (document-level chunking)
      - Long documents (>1500 chars) split into overlapping sub-chunks
      - Deduplication with order preservation
      - Output: chunks.pkl (35 unique chunks, avg 773 chars each)

    model-rag/build_faiss_index.py
      - Loads chunks.pkl
      - Encodes with all-MiniLM-L6-v2 (384-dimensional vectors)
      - Normalizes embeddings for cosine similarity
      - Builds FAISS IndexFlatIP (inner product = cosine for normalized vectors)
      - Output: faiss_index.bin (35 vectors, 384 dimensions)

    model-rag/rag_retriever.py
      - Loads FAISS index + chunks + embedding model (lazy singleton)
      - retrieve(query, top_k=5): returns top chunks with similarity scores
      - call_groq(prompt, context): calls Groq API with context-augmented prompt
      - ask_rag(question): full pipeline — retrieve + augment + generate
      - Reads GROQ_API_KEY from .env file automatically
      - CLI test mode: python rag_retriever.py "your question"
      - Interactive mode: python rag_retriever.py (then type questions)

    model-rag/run_pipeline.py
      - Orchestrator script that runs all steps in order:
          1. generate_knowledge_base.py
          2. create_chunks.py
          3. build_faiss_index.py
          4. (optional with --test) test retrieval with 5 sample queries
      - Usage: python run_pipeline.py --test

  PART 3: ISSUES FOUND IN ORIGINAL RAG & HOW THEY WERE FIXED
================================================================================

3.1 ISSUE #1: KNOWLEDGE BASE HAD ZERO DIVERSITY (CRITICAL)
    -------------------------------------------------------
    BEFORE:
      - generate_knowledge_base.py sampled 200 random failure rows
      - Every document used the EXACT same template:
          "Solar inverter anomaly detected.
           Observed telemetry signals:
           - inverter_power: [number]
           - pv1_power: [number]
           ...
           Interpretation:
           These conditions previously occurred before inverter failure.
           High temperature, unstable power output, and current variability
           are common indicators of inverter degradation."
      - Only 6 telemetry values changed between documents — everything else
        was identical text
      - No inverter IDs, no MACs, no plant identifiers
      - No alarm code explanations
      - No operational state meanings
      - No maintenance procedures
      - No root cause analysis
      - No thresholds or guidelines
      - Result: 200 documents that all said the same thing with different numbers

    AFTER:
      - 30 diverse documents covering 15 different content sections
      - Real per-inverter profiles with actual fleet statistics
      - Full alarm code reference with occurrence counts from the dataset
      - Full operational state reference with real counts
      - Root cause analysis for 6 common failure modes
      - Maintenance schedules, threshold guidelines, FAQ answers
      - Case studies from actual high-severity events in the data
      - Every document is unique and self-contained


3.2 ISSUE #2: CHUNKING WAS COMPLETELY BROKEN (CRITICAL)
    ----------------------------------------------------
    BEFORE:
      - create_chunks.py split the entire knowledge base by "\n" (newline)
      - This created 2,400 single-line fragments
      - Average chunk length: 33.7 characters (a single short sentence)
      - Example chunks:
          "- inverter_power: 43.2"
          "Solar inverter anomaly detected."
          "are common indicators of inverter degradation."
      - 1,353 of 2,400 chunks were DUPLICATES (56% duplication rate)
      - No chunk contained a complete thought or useful context

    AFTER:
      - Splits by "---DOC---" separator (document-level chunking)
      - Each chunk is a complete, self-contained paragraph
      - 35 unique chunks (zero duplicates)
      - Average chunk length: 773 characters (~15-20x longer)
      - Long documents (>1500 chars) split into overlapping sub-chunks
        with 200-char overlap for context continuity
      - Every chunk contains enough context to be useful on its own


3.3 ISSUE #3: RETRIEVAL RETURNED GARBAGE (CRITICAL)
    ------------------------------------------------
    BEFORE:
      - Because chunks were single lines, similarity search returned
        irrelevant fragments
      - Test queries and results:
          Q: "Why is inverter temperature increasing?"
          A: "- inverter_power: 43.2" (irrelevant)

          Q: "What alarm code 548 means?"
          A: "Solar inverter anomaly detected." (generic, wrong)

          Q: "Tell me about Plant 2"
          A: "- pv1_power: 125.8" (just a number)

          Q: "How to prevent failures?"
          A: "are common indicators of inverter degradation." (fragment)
      - Every query returned random line fragments instead of useful answers

    AFTER:
      - Retrieval returns complete, relevant documents
      - Test results with scores (cosine similarity, higher = better):
          Q: "Why is inverter temperature increasing?"
          → score=0.695 | FAQ: Rising inverter temperature causes... (RELEVANT)

          Q: "What does alarm code 548 mean?"
          → score=0.612 | Alarm code reference with full descriptions (RELEVANT)

          Q: "Tell me about Plant 2 performance"
          → score=0.426 | Comparative plant analysis (RELEVANT)

          Q: "How do I prevent inverter failures?"
          → score=0.615 | Case studies + maintenance recommendations (RELEVANT)

          Q: "What is op state 5120?"
          → score=0.500 | Op state reference + FAQ answer (RELEVANT)


3.4 ISSUE #4: NO LLM INTEGRATION (INCOMPLETE)
    -------------------------------------------
    BEFORE:
      - rag_retriver.py (note: typo in filename) retrieved chunks and
        built a prompt string, but NEVER called any LLM
      - The create_prompt() function generated a prompt template but
        just printed it to console
      - No Groq, no OpenAI, no local model — nothing
      - The RAG pipeline was literally just "retrieve and print"

    AFTER:
      - rag_retriever.py (filename fixed) has full Groq API integration
      - Uses llama-3.3-70b-versatile model
      - System prompt configured for Sunlytix solar monitoring context
      - Reads GROQ_API_KEY from .env file automatically
      - Proper error handling for API failures
      - Returns structured response: { answer: str, sources: list }
      - Interactive CLI test mode for easy debugging
      - Can be imported as a module: from rag_retriever import ask_rag


3.5 ISSUE #5: FAISS INDEX USED L2 DISTANCE (SUBOPTIMAL)
    ----------------------------------------------------
    BEFORE:
      - Used IndexFlatL2 (Euclidean distance) without normalized embeddings
      - L2 distance doesn't account for vector magnitude differences
      - Results were less accurate for semantic similarity

    AFTER:
      - Embeddings are normalized (normalize_embeddings=True)
      - Uses IndexFlatIP (Inner Product) which equals cosine similarity
        for normalized vectors
      - Produces more accurate semantic matching scores (0-1 range)


3.6 ISSUE #6: NO PIPELINE ORCHESTRATION
    ------------------------------------
    BEFORE:
      - Each script had to be run manually in the correct order
      - No way to rebuild everything with one command
      - No validation or testing step

    AFTER:
      - run_pipeline.py orchestrates all steps:
          1. Generate knowledge base
          2. Create chunks
          3. Build FAISS index
          4. (optional) Test retrieval
      - Single command: python run_pipeline.py --test
      - Reports timing for each step
      - Exits immediately on failure

================================================================================
  PART 5: HOW TO USE
================================================================================

  REBUILD RAG (after data changes):
    cd model-rag
    python run_pipeline.py --test

  TEST A SINGLE QUESTION:
    python rag_retriever.py "Why is inverter temperature high?"

  INTERACTIVE MODE:
    python rag_retriever.py
    > Type questions, get answers
    > Type 'quit' to exit

  USE IN CODE:
    from rag_retriever import ask_rag
    result = ask_rag("What does alarm 548 mean?")
    print(result["answer"])    # LLM-generated answer
    print(result["sources"])   # retrieved chunks with scores

  RUN PREDICTION:
    python predict.py
    # Reads new_telemetry_data.csv, outputs prediction + probability

  RUN SHAP EXPLANATION:
    python exp_model.py
    # Shows top 5 features driving the prediction

  REQUIREMENTS:
    pip install pandas numpy scikit-learn faiss-cpu sentence-transformers shap

  GROQ API KEY:
    Set GROQ_API_KEY in your .env file for LLM-powered answers.
    Get a key at: https://console.groq.com/keys
